# src/core/types

В этой папке описываются общие типы и модели данных приложения.

Модель данных — это описание сущности проекта: какие поля у неё есть, какие значения допустимы и как эта сущность связана с другими частями приложения.

Типы из этой папки используются в UI, core-модулях и будущей автоматизации, чтобы все части проекта работали с данными в одном формате.

## Provider

Описание AI-сервиса.

Например:

- Kling
- Pika
- Hailuo
- ChatGPT
- Gemini

Provider описывает сам сервис, а не конкретный аккаунт пользователя.

Примерные данные:

- id сервиса;
- название;
- категория;
- адрес сайта;
- адрес страницы запуска;
- есть ли бесплатные кредиты;
- период начисления кредитов;
- тип получения кредитов.

## Account

Описание подключенного аккаунта пользователя.

Один аккаунт всегда связан с одним Provider.

Например:

- kling-main;
- kling-work;
- pika-personal.

У одного AI-сервиса может быть несколько аккаунтов.

Примерные данные:

- id аккаунта;
- id сервиса;
- отображаемое название;
- статус аккаунта;
- путь к browser profile;
- дата последней проверки.

## Account Status

Статус подключенного аккаунта.

Нужен, чтобы приложение понимало, что сейчас происходит с аккаунтом.

Возможные статусы:

- not_connected — аккаунт ещё не подключен;
- connected — аккаунт подключен;
- session_expired — сессия истекла, нужен повторный вход;
- manual_required — требуется ручное действие пользователя;
- error — произошла ошибка.

## Browser Profile

Отдельный браузерный профиль, привязанный к аккаунту.

Главное правило: один аккаунт = один browser profile.
Это нужно, чтобы разные аккаунты одного сервиса не смешивались между собой.

Browser Profile будет использоваться Playwright для открытия сайта с уже сохранённой сессией пользователя.

## Daily Check

Результат проверки аккаунта или сервиса.

Например, приложение проверило Kling и сохранило:

- дата проверки;
- id аккаунта;
- статус проверки;
- количество кредитов, если удалось прочитать;
- сообщение об ошибке, если что-то пошло не так.

Daily Check нужен для истории проверок и будущего Dashboard.

## Automation Step

Один шаг сценария автоматизации.

Например:

- открыть сайт;
- подождать загрузку страницы;
- проверить, нужен ли логин;
- дождаться ручного действия пользователя;
- прочитать текст с количеством кредитов;
- нажать кнопку получения ежедневных кредитов.

Automation Step не должен описывать конкретный код Playwright.

Он описывает сценарий на уровне данных, чтобы позже automation engine мог выполнять эти шаги.
